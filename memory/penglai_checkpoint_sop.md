# 符号化 Checkpoint SOP（长任务断点续传）

触发场景：任务预计 >20 轮 / 产生超长输出(日志/抓取结果/报告 >100 行) / 用户说"慢慢做、做完叫我"。

## 核心原则：工作记忆只存指针，不存原文
LLM 上下文是稀缺资源。长内容一律落盘，checkpoint 里只留「文件路径 + 3 行结论 + 下一步」。

## 操作
1. **长输出落盘**：超过 100 行的命令输出/抓取内容，写入 `./task_<主题>_<日期>.log`（cwd=temp 下），
   不要整段贴进回复或 key_info。
2. **符号化 checkpoint**：`update_working_checkpoint` 的 key_info 按此模板写：
   ```
   [目标] 一句话
   [进度] 阶段 N/M：已完成…，当前在…
   [指针] 详细数据→./task_xxx.log；中间产物→./xxx/
   [下一步] 动词开头的一句话
   [坑] 本任务已踩过的坑（防重蹈）
   ```
3. **复杂流程画图**：步骤 >5 且有分支依赖时，把流程写成 Mermaid 图落盘
   `./task_<主题>_plan.md`，节点上标 ✅/🔄/⬜，恢复时先读图。
4. **恢复**：中断后续做时，先 file_read checkpoint 里的指针文件，再继续；禁止凭记忆重做。

## 坑
- 把整段日志塞进 key_info → token 爆炸且每轮复利，必犯。
- 落盘文件起名无主题无日期 → 恢复时找不到。
- 只存指针不存结论 → 每次恢复都要重读全文；指针旁必须带 3 行内结论。
- 本 SOP 管"任务内"临时记忆；任务结束后值得长存的经验走 L0 记忆 SOP 结算，别混淆。
